# Changelog

## [v1.6.3] - 2026-06-07

> Production-hardening release. Focused on three pillars: correctness fixes to
> the core recovery/circuit-breaker infrastructure; end-to-end polish of all SQL
> database plugins (health-check, metrics, DSN parsing); and a significant
> robustness overhaul of the `lynx` CLI ‚Äî especially the `plugin` subcommand
> which now correctly manages Go blank imports. No breaking API changes.

### ‚ú® Features

**Core framework**
- **Concrete recovery strategies**: `DefaultRecoveryStrategy` (no-op stub) is
  replaced by three real implementations ‚Äî `RetryRecoveryStrategy` (exponential
  back-off with ¬±25 % jitter, up to 3 retries), `RestartRecoveryStrategy`
  (cooldown + optional restart hook), and `FallbackRecoveryStrategy` (immediate
  degradation). All are exported on the `lynx` facade as type aliases so existing
  call-sites keep compiling.
- **`ActionFunc` hook**: all three new strategies accept an `ActionFunc(ctx,
  record) error` so callers can inject custom recovery logic without subclassing.

**CLI ‚Äî `plugin` command**
- **Blank import management**: `lynx plugin install` now writes (or updates)
  `cmd/<svc>/plugins_gen.go` with the blank import that registers the plugin's
  `init()`. Without this the plugin was silently never loaded. `lynx plugin
  remove` removes the same import *before* calling `go get @none`, so
  `go mod tidy` can no longer re-add the dropped dependency.
- **Works outside a project directory**: `plugin list`, `plugin search`, and
  `plugin info` no longer require a `go.mod` to be present. They use
  `NewReadOnlyPluginManager` which attempts project root detection but degrades
  gracefully, showing all plugins as "Available" when called outside a project.
- **Config path aligned with `lynx new`**: plugin config templates are now
  written to `configs/<name>.yaml` (matching the layout generated by `lynx new`)
  rather than `conf/<name>.yaml`.
- **Stale cache fallback**: when GitHub is unreachable and a disk cache exists
  (even an expired one), the plugin registry falls back to the stale cache with
  a warning instead of failing the command entirely.

### üêõ Fixes

**Core framework**
- **`CircuitBreaker` HalfOpen single-probe guarantee**: multiple goroutines
  racing on a freshly-expired timeout could all transition to `HalfOpen` and
  each launch an independent probe. An `int32 probing` atomic flag now ensures
  exactly one goroutine wins the probe; all others are rejected until the probe
  completes. Three new tests verify this under `-race` with 64 concurrent
  goroutines.
- **`ErrorRecoveryManager` lock contention**: `getStackTrace` (calls
  `runtime.Callers`) and the recovery goroutine launch were both inside the
  write lock in `RecordError`. Stack capture is now done before acquiring the
  lock; the goroutine is launched after an explicit `Unlock()`.
- **`appShutdownHook` data race**: the package-level `appShutdownHook` function
  variable was read in `close()` and written in `SetAppShutdownHook()` without
  synchronisation. A `sync.RWMutex` guard and an `invokeShutdownHook()` helper
  prevent the race.

**SQL plugins (`lynx-sql-sdk`, `lynx-mysql`, `lynx-pgsql`, `lynx-mssql`)**
- **`CheckHealthContext` double-reconnect**: the health check called
  `GetDBWithContext` which itself triggers reconnect, then the health check path
  also triggered a reconnect ‚Äî two reconnects per failed health check. The fix
  reads `p.db` directly under `RLock`, bypassing `GetDBWithContext`.
- **`CheckHealthContext` hardcoded 5 s timeout**: both timeout values inside the
  function were hardcoded `5 * time.Second`; they now use
  `config.ConnectTimeout` so operator configuration is respected.
- **mssql missing failure metric**: `DBMssqlClient.CheckHealth` never called
  `RecordHealthCheck(false, ...)` on error, so Prometheus health-check failure
  counters were always zero.
- **PostgreSQL key-value DSN quoted values**: `extractInstance` and
  `extractDatabaseName` in `lynx-pgsql` now use a proper parser
  (`parseKVDSN`) that handles single-quoted values with embedded spaces and
  backslash escapes (e.g. `password='my pass'`).
- **MySQL DSN format coverage**: `extractInstance` in `lynx-mysql` now handles
  all four Go driver DSN forms: `user:pass@tcp(host:port)/db`,
  `@unix(/tmp/mysql.sock)/db`, bare `host:port/db`, and no-`@` forms.

**CLI ‚Äî `run` command**
- **Watcher goroutine leak**: `NewFileWatcher` starts an internal polling
  goroutine. `Runner.Start` never called `watcher.Stop()`, so the goroutine
  ran forever after the process exited. It is now stopped after `ctx` cancels.
- **`--run-args` space handling**: `strings.Fields` was used to split runtime
  arguments, breaking any argument containing spaces. Switched to the same
  `parseBuildArgs` parser used by `--build-args`.
- **Double `Wait` on same `exec.Cmd`**: the crash-monitor goroutine in `run()`
  and the graceful-stop path in `stop()` both called `process.Wait()`. `Wait`
  must only be called once. A `processDone chan struct{}` channel (closed by the
  monitor goroutine) is now used by `stop()` instead.
- **`--skip-build` binary check**: passing `--skip-build` when no binary exists
  at the expected path now returns a clear error instead of crashing on exec.

**CLI ‚Äî `doctor` command**
- **`GoModCheck` false WARNING for user projects**: the check previously warned
  for any module path not under `github.com/go-lynx/`. Projects created by
  `lynx new` have their own module paths and were always flagged. Now any
  parseable `go.mod` is `OK`.
- **`--format` early validation**: an invalid `--format` value used to silently
  fall back to text after running all checks. It is now validated before the
  checks run.

**CLI ‚Äî `plugin` command**
- **`plugin remove` wrong config path**: `RemovePlugin` attempted to delete
  `conf/<name>.yaml` relative to CWD, while `InstallPlugin` creates the file at
  `<projectRoot>/configs/<name>.yaml`. The path is now consistent.
- **`plugin list --type` silent empty result**: an unrecognised type string
  returned an empty list with no diagnostic. The value is now validated up-front
  with a descriptive error.

### ‚úÖ Tests

- `TestCircuitBreaker_HalfOpen_SingleProbe`: 64 goroutines race on an expired
  timeout; asserts exactly 1 probe is allowed (run with `-race`).
- `TestCircuitBreaker_HalfOpen_ProbeSuccessUnblocksCircuit`: probe success
  transitions to Closed and unblocks subsequent callers.
- `TestCircuitBreaker_HalfOpen_ProbeFailureReturnsOpen`: failed probe reopens
  the circuit; a fresh probe is allowed after the next timeout.
- `TestParseKVDSN` (lynx-pgsql): 6 cases covering empty, plain, quoted-value,
  embedded-space, and backslash-escape DSN forms.
- `TestExtractInstanceAndDatabase_Pgsql` (lynx-pgsql): 5 cases for both URL
  and key-value DSN formats.
- `TestExtractInstance_MySQL` (lynx-mysql): 5 cases for tcp, unix socket, bare,
  no-`@`, and URL forms.

### ‚¨ÜÔ∏è Upgrade Notes

No API surface changes; all existing code compiles without modification.

- If you registered a `DefaultRecoveryStrategy` directly and relied on its
  no-op behaviour, replace it with `NewFallbackRecoveryStrategy` which
  provides an equivalent immediate-degradation path.
- `lynx plugin install` now writes `cmd/<svc>/plugins_gen.go`. If you manage
  blank imports manually, check this file to avoid duplicates.
- Config templates from `lynx plugin install` are now written to `configs/`
  (not `conf/`). Remove any old `conf/<plugin>.yaml` files generated by prior
  versions of the CLI if they are no longer needed.

## [v1.6.2] - 2026-06-04

> Stabilization release on the `v1.6.x` beta line. Focused on making plugin
> lifecycle cancellation real, fixing two panics in version handling, and
> raising core test coverage. No breaking API changes.

### ‚ú® Features

- **Genuinely cancellable plugin lifecycle**: `StartContext`, `StopContext`, and
  `InitializeContext` on the base plugin no longer wrap the synchronous lifecycle
  in a goroutine and merely race `ctx.Done()` (which leaked the goroutine and
  could flip the plugin back to `StatusActive` behind the manager's back). They
  now run framework phases inline with `ctx` checks at every boundary, prefer the
  plugin's own context-aware work, and re-check `ctx` so a late-completing legacy
  task can never promote a cancelled plugin to `Active`.
- **Context-aware lifecycle step hooks**: new optional interfaces
  `ContextResourceInitializer`, `ContextStartupTasker`, and `ContextCleanupTasker`.
  A plugin that implements any of them gets its own work cancelled for real (the
  hook receives the lifecycle `ctx`). Plugins that only implement the legacy
  non-context steps are abandoned **safely** on timeout: the orphaned goroutine
  never mutates plugin status, and is counted and logged (`OrphanedStageCount`).
- **Automatic context-aware routing**: `HasTrueContextLifecycle` now treats a
  plugin as genuinely cancellable when it exposes the `LifecycleWithContext`
  entrypoints and implements at least one context step hook ‚Äî no explicit
  `IsContextAware()` / `PluginProtocol().ContextLifecycle` opt-in required. The
  manager routes such plugins through the context-aware path automatically.

### üêõ Fixes

- **`compareVersionNumeric` panic**: comparing two versions with a different number
  of components (e.g. `"1.2"` vs `"1.2.1"`) indexed a slice before the bounds
  check, causing an index-out-of-range panic. Bounds are now checked first.
- **`VersionManager.ParseVersion` panic**: parsing any version without `+build`
  metadata (i.e. almost all versions, including `"1.2.3"`) read `buildParts[1]`
  unconditionally and panicked. Build metadata is now captured only when present.

### ‚úÖ Tests

- Core `plugins` package coverage raised from ~30% to ~72%, concentrated on the
  dependency graph, version handling, lifecycle state machine, conflict resolver,
  runtime/resource layer, and error handling.
- Added concurrency/chaos suites (run under `-race`) for the lifecycle state
  machine, cancellation storms, legacy orphan accounting, and the manager's
  concurrent start/stop/init under timeouts and panics.

## [v1.6.0] - 2026-04-02

> Draft release notes for the stable `v1.6.0` cut. This section is intentionally limited to landed changes with evidence in the current workspace. It does not mark `P1-3`, `P2-2`, `P2-3`, or `P2-4` as complete.

### üéØ Breaking Changes

- **`lynx-layout` bootstrap now returns startup errors instead of panicking**: `server.NewGRPCServer` and `server.NewHTTPServer` now return `(*Server, error)`, and the generated bootstrap path propagates those errors to `main`. Projects that relied on panic-on-registration behavior must switch to normal error handling.
- **`lynx-redis` deprecated lifetime aliases are now enforced consistently**: `idle_timeout` and `max_conn_age` are treated as compatibility aliases for `conn_max_idle_time` and `conn_max_lifetime`. If both the legacy and preferred fields are set, they must match. Configurations that previously relied on inconsistent mixed values may now fail validation fast.

---

### ‚ú® Features

- **Template login auth now has a real outbound gRPC skeleton**: `lynx-layout` no longer leaves `LoginAuth()` as an empty-token stub. The template now validates input, propagates `context`, reads `lynx.layout.auth.grpc.*` or `LYNX_LAYOUT_AUTH_GRPC_*`, invokes a configured downstream gRPC method, and extracts a token from the response payload. Until a concrete auth proto is wired in, the scaffold uses a generic structured payload so projects can plug in their real auth service without rewriting the call path.
- **Core packages completed the remaining `interface{}` -> `any` cleanup**: the remaining `lynx/` code, tests, and relevant docs now use `any`, aligning the codebase with the typed event/runtime work already merged and removing the last old-style empty-interface usages in the core tree.
- **Startup failure propagation is now explicit across nacos/etcd/kafka**: `lynx-nacos` verifies connectivity before declaring startup success, `lynx-etcd` actively probes connectivity instead of trusting lazy client creation, and `lynx-kafka` returns broker connectivity failures instead of logging and continuing in a disconnected state. This is the shipped core of `P1-4`; broader cross-plugin integration coverage remains a separate follow-up task.

---

### üêõ Fixes

- **`lynx-layout` server initialization now exits via the normal error path**: gRPC/HTTP getter failures and service registration panics are converted into returned errors, so startup can fail cleanly without bypassing graceful shutdown handling.
- **`lynx-layout` login auth failures are no longer silent**: missing config, invalid request state, empty token responses, or downstream gRPC failures now surface as explicit errors instead of `"" , nil`.
- **`lynx-redis` connection-pool field mapping now matches go-redis semantics**: `max_idle_conns`, `idle_timeout`, and `max_conn_age` are wired to the intended runtime options, `conn_max_lifetime` is present in generated config, and validation/defaulting logic now covers the compatibility aliases instead of leaving `ConnMaxLifetime` mismatched or ineffective.

---

### üöö Beta Exit / Upgrade Notes

- **Stable-release draft scope**: this draft covers landed work for `P0-1`, `P0-2`, `P1-1`, `P1-2`, and `P1-4` only. It intentionally does not promote `P1-3`, `P2-2`, `P2-3`, or `P2-4` to done.
- **Completed verification evidence**
  - `lynx`: `go build ./...`
  - `lynx`: `go test ./plugins ./subscribe ./pkg/auth/jwt ./log ./pkg/errx`
  - `lynx-redis`: `go test ./... -run 'Test(BuildUniversalOptions|ValidateRedisConfig|ValidateAndSetDefaults|SetDefaultValues)'`
  - `lynx-layout`: `gofmt` and `go generate ./cmd/user`
- **Still blocked before a final stable-release sign-off**
  - `lynx`: `go test ./plugins/... ./...` is still blocked by pre-existing `cache` and `tls` duplicate test names, plus current `boot` and `events` test failures.
  - `lynx-layout`: `go test ./...` is still blocked by the pre-existing sibling-module `../lynx-redis` compile issue that was discovered while validating the auth/server fixes.
  - `P1-4`: source-level changes are landed, and `lynx-etcd/lifecycle_test.go` already covers the unreachable-endpoint startup failure path, but wider cross-plugin regression coverage still depends on the pending `P1-3` integration-test task.
- **Upgrade checklist from `v1.6.0-beta`**
  - Handle returned startup errors in custom `lynx-layout` bootstrap code instead of assuming panic-driven failure.
  - Normalize Redis config to `conn_max_idle_time` / `conn_max_lifetime`; do not keep conflicting deprecated aliases in the same config block.
  - If you adopt the updated login template, set `lynx.layout.auth.grpc.service`, `lynx.layout.auth.grpc.method`, and optional timeout/env overrides before expecting non-empty login tokens.

## Upgrade Guide

### Upgrading from v1.2.3 to v1.5.0-beta

1. **Update Go Version**
   ```bash
   # Ensure Go 1.25+ is installed
   go version
   ```

2. **Update Dependencies**
   ```bash
   go get github.com/go-lynx/lynx@v1.5.0-beta
   go mod tidy
   ```

3. **Plugin Migration**
   - Plugins are now in separate Git repositories
   - Update plugin import paths if necessary

4. **Configuration Updates**
   - Review gRPC configuration for message size limits
   - Ensure TLS configuration meets new security requirements

5. **API Changes**
   - Migrate from `PluginManager` to `TypedPluginManager` if applicable
   - Check error handling: some functions now return errors instead of panicking
